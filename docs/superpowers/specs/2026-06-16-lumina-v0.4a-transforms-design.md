# Lumina v0.4a — Трансформации (поворот / отражение) — дизайн

Дата: 2026-06-16. Фаза: v0.4a (первая половина v0.4; EXIF popup/запись — отдельной под-фазой v0.4b).
Спецификация проекта — `SPEC_lumina.md` (§5, §6). Роадмап — `ROADMAP.md`.

## Контекст и решение по охвату

v0.4 в роадмапе объединяет два почти независимых блока — **трансформации** (поворот/отражение)
и **EXIF** (чтение/popup/запись). Они имеют разный уровень риска: запись EXIF тянет тяжёлую нативную
зависимость (ExifTool FFI на MSVC). Поэтому v0.4 разбита:

- **v0.4a (этот документ)** — трансформации. Чистый Rust, минимум новых зависимостей (`kamadak-exif` —
  pure-Rust, лёгкая), без записи на диск.
- **v0.4b (позже)** — чтение остальных EXIF-тегов, EXIF popup с инлайн-редактированием, запись на диск
  (ExifTool / XMP-sidecar / lossless JFIF-rotation).

### Что входит в v0.4a

1. **Авто-ориентация по EXIF `Orientation`** при открытии — это не фича, а корректность отображения:
   телефонные JPEG и многие камерные файлы хранят «правильный» поворот в EXIF-теге, и без его учёта
   портретные фото показываются «на боку». Сейчас это фактический баг (просто не всплыл на тестовых RAF).
2. **Ручной поворот/отражение** пользователем (`R`/`Shift+R`/`H`/`V`, кнопка поворота в bottom bar, `Ctrl+Z`).
3. **Сессионная память** ручного transform по пути файла. На диск ничего не пишется.

### Что НЕ входит (→ v0.4b)

Запись на диск, обновление EXIF `Orientation`, XMP-sidecar для RAW, lossless JFIF-rotation для JPEG,
ExifTool, кнопка «сохранить поворот». Чтение прочих EXIF-тегов (Make/Model/ExposureTime/…) и popup.

## 1. Ключевая идея: два разных слоя поворота

EXIF-ориентация и ручной поворот **не смешиваются** в одной матрице. Они разведены по уровням:

- **Авто-ориентация по EXIF — нормализуется в слое декодера.** Каждый декодер обязан вернуть
  `DecodedImage` уже в «вертикальной» (upright) ориентации. Это разовая операция на rayon-потоке декода.
  Результат: вся остальная система всегда считает изображение правильно ориентированным, и риск двойного
  поворота исключён в одном месте.
- **Ручной поворот/отражение — в матрице вида (`view.rs`).** Живой, мгновенный, без передекода.
  Стартует с identity для каждого фото; `Ctrl+Z` откатывает к identity (то есть к EXIF-нормализованному виду).

Выгода разделения: `view.rs` ничего не знает про EXIF; декодеры ничего не знают про ручной поворот.
Каждый слой тестируется изолированно.

### Обоснование различия механизмов

- EXIF-ориентация применяется **разово** при декоде, на фоновом потоке, средствами `image` — поэтому дешевле
  всего «запечь» её в пиксели (нормализовать буфер). Это также делает `DecodedImage` самодостаточным.
- Ручной поворот **интерактивный** и должен быть мгновенным без передекода — поэтому он живёт в матрице вида
  (GPU-дёшево, как и текущий zoom/pan).

## 2. Компоненты и поток данных

### 2.1 `src/exif/mod.rs` (новый модуль, минимальный для v0.4a)

```rust
/// EXIF Orientation (TIFF tag 0x0112), значения 1..=8.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Default)]
pub enum Orientation {
    #[default]
    Normal,            // 1
    FlipH,             // 2
    Rotate180,         // 3
    FlipV,             // 4
    Transpose,         // 5  (отражение по диагонали)
    Rotate90,          // 6  (поворот по часовой 90°)
    Transverse,        // 7
    Rotate270,         // 8  (поворот по часовой 270° / против часовой 90°)
}

impl Orientation {
    pub fn from_exif_u16(v: u16) -> Self; // неизвестное/0 → Normal
}

/// Прочитать Orientation из EXIF файла. Для форматов без EXIF / без тега → Normal.
/// Используется только StandardDecoder (JPEG/TIFF/WebP); RAW/HEIC берут ориентацию иначе (см. 2.2).
pub fn read_orientation(path: &Path) -> Orientation;
```

Зависимость: `kamadak-exif` (pure-Rust). Добавляется в `Cargo.toml` в v0.4a.

### 2.2 Декодеры — нормализация до upright

Каждый декодер гарантирует upright-выход. Способ зависит от формата:

| Декодер | Источник ориентации | Действие |
|---|---|---|
| `StandardDecoder` (JPEG/TIFF/WebP/…) | `exif::read_orientation(path)` | Применить поворот/отражение к `DynamicImage` средствами `image` **до** `to_rgba8`. `image` сам ориентацию НЕ применяет. |
| `HeicDecoder` (HEIC/HEIF) | libheif применяет `irot`/`imir` при декоде сам | Ничего не делаем (повторно нельзя). Добавить тест-подтверждение, что выход upright. |
| `RawDecoder` (RAF/NEF/ARW/…) | метаданные rawler / встроенный JPEG | Привести **обе** стадии (embedded preview и develop) к одной upright-ориентации. Приёмка на реальных файлах (риск двойного поворота). |

Реализационные заметки:
- **Standard:** `image` 0.25 умеет применить ориентацию (`DynamicImage::apply_orientation` /
  `image::imageops` rotate/flip). Маппинг `Orientation` → последовательность операций — в новом модуле,
  чтобы не дублировать в декодере.
- **RAW:** встроенный JPEG камеры может быть уже ориентирован, а `RawDevelop` — нет (или наоборот);
  нужно выяснить на образцах и привести обе стадии к одному результату. Двухстадийный конвейер preview→full
  не должен «прыгать» по ориентации между кадрами.
- Известное ограничение остаётся: формат без распознаваемой ориентации трактуется как `Normal`.

### 2.3 `view.rs` — ручной `Transform`

Поля `rotation: u8`, `flip_h: bool`, `flip_v: bool` уже объявлены (задел) — задействуем в `matrix()`.

- `rotation` хранит угол по часовой ∈ {0, 90, 180, 270}.
- **Центрированная формула матрицы** (заменяет текущую угловую): unit-quad [0,1] центрируется,
  поворачивается, отражается, масштабируется и переносится в центр окна + pan:

  ```
  model = T(win_center + pan) · R(rotation) · Flip(flip_h, flip_v) · S(img·zoom) · T(-0.5, -0.5)
  ```

  где `img` — исходные (не повёрнутые) размеры изображения. Текущий blit-пайплайн (quad 0..1 + uv)
  при этом не меняется — вся правка в построении матрицы.
- **Эффективные размеры:** новый хелпер `effective_dims(img, rotation)` свопает ширину/высоту для
  90°/270°. `fit_zoom`, `clamp_pan` и `min_zoom` считаются по эффективным размерам (повёрнутая картинка
  вписывается в окно по своему фактическому габариту).
- API дополняется методами: `rotate_cw()`, `rotate_ccw()`, `flip_horizontal()`, `flip_vertical()`,
  `reset_transform()` (для `Ctrl+Z`), `transform() -> Transform` / `set_transform(Transform)`
  (для сессионной памяти).

### 2.4 `app.rs` — сессионная память и интеграция

- `transforms: HashMap<PathBuf, Transform>` в состоянии приложения.
- При открытии фото: восстановить сохранённый ручной transform по пути, иначе identity.
- При любой ручной трансформации: обновить запись в HashMap, вызвать `clamp_pan`, и если вид в `is_fit` —
  пересчитать `fit_zoom`/`min_zoom` под новые эффективные размеры (фото остаётся вписанным).
- Память живёт только на время сессии (не сериализуется).

## 3. Ввод и UI

| Клавиша / действие | Результат |
|---|---|
| `R` | Поворот по часовой +90° (mod 360) |
| `Shift+R` | Поворот против часовой −90° |
| `H` | Горизонтальное отражение (toggle) |
| `V` | Вертикальное отражение (toggle) |
| `Ctrl+Z` | Откат ручного transform текущего фото к identity |
| Кнопка поворота в bottom bar | То же, что `R` (сейчас no-op → оживить) |

- Диспетчинг — в `app.rs::window_event`, по образцу существующих стрелок/Ctrl+0/Ctrl+1.
- `Ctrl+Z` в v0.4a сбрасывает к identity (полный откат ручных операций текущего фото). Пошаговый
  undo-стек — возможное улучшение, но для v0.4a достаточно сброса (YAGNI).
- Кнопка поворота уже отрисована (Tabler icon) — добавляется только обработчик клика.

## 4. Обработка крайних случаев

- **Поворот + zoom:** опорная точка zoom (`zoom_at`) берётся относительно центра окна — не зависит от
  поворота, остаётся корректной.
- **Поворот в режиме fit:** после поворота при `is_fit` пересчитываем fit по эффективным размерам, чтобы
  повёрнутое фото вписалось (иначе портрет после поворота на 90° вылезет за окно).
- **Смена preview→full при активном ручном повороте:** `rescale_for_new_image` сохраняет экранный размер;
  ручной transform применяется к обеим стадиям одинаково (хранится отдельно от размеров).
- **Файл без EXIF / неизвестная ориентация:** `Normal` (identity).
- **HEIC/RAW уже ориентированы декодером:** ручной transform всё равно стартует с identity — двойного
  поворота нет, т.к. EXIF-ориентация запечена в пиксели, а не в матрицу.

## 5. Тестирование

**Чистые юниты (`view.rs`):**
- `matrix()` конечна для всех 4 поворотов × 4 комбинаций flip.
- `effective_dims` свопает W/H для 90°/270°, сохраняет для 0°/180°.
- `fit_zoom` по эффективным размерам: портрет 90° вписывается по новому габариту.
- `clamp_pan` корректен с учётом свопа размеров.
- `rotate_cw`/`ccw` цикличны (4 поворота возвращают к исходному); `reset_transform` → identity.

**Чистые юниты (`exif`):**
- Маппинг всех 8 значений `Orientation` → ожидаемый `Transform`.
- `from_exif_u16` для 0/неизвестных → `Normal`.

**Декодеры:**
- `StandardDecoder`: мини-фикстура JPEG с тегом `Orientation=6` декодится повёрнутой правильно
  (генерируется через `make_fixtures`).
- `HeicDecoder` / `RawDecoder`: `#[ignore]`-приёмка на реальных образцах (проверка отсутствия двойного
  поворота).
