# Lumina v0.4c — Навигация и эргономика ввода — дизайн

**Дата:** 2026-06-16
**Статус:** одобрен к реализации

## Цель

Три независимые эргономические фичи ввода/навигации, без записи на диск:

1. **Раскладко-независимые шорткаты** — `R`/`H`/`V`/`F`/`Ctrl+Z`/`Ctrl+0`/`Ctrl+1` работают на любой раскладке клавиатуры (кириллица К/Р/М/… и др.), потому что матчатся по физической клавише, а не по символу.
2. **Экранные стрелки ‹ ›** — тонкие вертикальные полосы у левого/правого края области просмотра (оконный режим), во всю высоту, проявляются при наведении курсора; клик листает фото.
3. **Свайп drag-жестом** — перелистывание горизонтальным перетаскиванием, активно **только** когда фото показано в размере по умолчанию (fit, без зума) — чтобы не конфликтовать с drag-pan при зуме. Фото следует за курсором, на отпускании за порогом листает, иначе плавно откатывается.

**Вне scope** (отдельный спек, v0.4b): запись поворота в EXIF Orientation оригинальных файлов (для поворота превью в Проводнике Windows).

## Архитектурный принцип

Сохраняем разделение проекта: **чистое ядро** (`src/input/`, `src/ui/{layout,hit,scene}`, `src/view.rs`) тестируется юнитами без GPU/winit; диспетчинг событий в `app.rs` — тонкая обёртка, исполняющая решения чистого ядра. Каждая фича добавляет логику в чистое ядро и минимальную интеграцию в `app.rs`.

## #1 — Раскладко-независимые шорткаты

### Подход
winit отдаёт в `KeyEvent` как `logical_key` (символ с учётом раскладки — на кириллице «к» вместо «r»), так и `physical_key` (`PhysicalKey::Code(KeyCode)` — позиция клавиши, не зависит от раскладки). Переключаем диспетчинг трансформаций/fullscreen/зум-пресетов на `physical_key`.

Маппинг (физическая клавиша → действие):
- `KeyCode::KeyR` → поворот: `Shift` ⇒ против часовой, иначе по часовой
- `KeyCode::KeyH` → отражение по горизонтали
- `KeyCode::KeyV` → отражение по вертикали
- `KeyCode::KeyF` (без `Ctrl`) → fullscreen
- `Ctrl + KeyCode::KeyZ` → сброс ручной трансформации
- `Ctrl + KeyCode::Digit0` → fit, `Ctrl + KeyCode::Digit1` → 100%

`NamedKey` (`ArrowLeft/Right`, `Home`, `End`, `F11`, `Escape`) уже не зависят от раскладки — остаются как есть.

### Чистое ядро
В `src/input/` вводим:
```rust
pub enum Action {
    RotateCw, RotateCcw, FlipH, FlipV, ResetTransform,
    ToggleFullscreen, FitView, ActualSize,
}
pub fn map_key(code: KeyCode, ctrl: bool, shift: bool) -> Option<Action>;
```
`map_key` — единственная точка истины маппинга, покрыта юнит-тестами (все коды, комбинации с ctrl/shift, отсутствие действия для неизвестных). `app.rs` в `KeyboardInput` берёт `code` из `event.physical_key` (если `PhysicalKey::Code`), зовёт `map_key`, исполняет `Action` существующими методами (`rotate_cw`, `after_transform_change`, `toggle_fullscreen`, …).

### Интеграция
`app.rs::window_event` ветка `KeyboardInput`: заменяем матчинг `logical_key` Character на `physical_key` + `map_key`. `F11`/`Escape`/стрелки/Home/End — прежний матчинг по `NamedKey`. Порядок сохраняется: NamedKey-перехваты → `map_key`-действия → навигация.

### Тесты
`input::map_key`: `KeyR`→RotateCw, `Shift+KeyR`→RotateCcw, `KeyH`→FlipH, `KeyV`→FlipV, `KeyF`→ToggleFullscreen, `Ctrl+KeyF`→None (не fullscreen под ctrl), `Ctrl+KeyZ`→ResetTransform, `KeyZ` без ctrl→None, `Ctrl+Digit0`→FitView, `Ctrl+Digit1`→ActualSize, `KeyA`→None.

## #3 — Экранные стрелки ‹ ›

### Геометрия (`src/ui/layout.rs`)
Добавляем в `UiLayout` поля `nav_prev: Rect`, `nav_next: Rect`.
- Ширина полосы `NAV_ARROW_W` (новая константа темы, ~44 лог.px × scale).
- Высота = вся высота viewer-региона (`viewer.y .. viewer.y + viewer.h`).
- `nav_prev` прижата к левому краю (`x = 0`), `nav_next` — к правому (`x = win.x - NAV_ARROW_W`).
- В fullscreen — `ZERO` (как прочий оконный хром): стрелок нет, согласно требованию.

### Hit (`src/ui/hit.rs`)
Новые регионы `Region::NavPrev`, `Region::NavNext`. В `hit()` проверяются **после** краёв ресайза (крайние `RESIZE_BORDER`=6px остаются под resize — «почти у самого края»), но до `Region::None`. Полосы лежат в viewer-области (ниже titlebar, выше divider), поэтому не конфликтуют с titlebar/bottom bar.

### Состояние и проявление (`src/ui/scene.rs`, `app.rs`)
В `UiState`:
- `nav_alpha: [f32; 2]` — прозрачность [prev, next], анимируется к 1 при hover соответствующей полосы, иначе к 0 (лерп в `RedrawRequested`, как `bottom_factor`).
- `can_prev: bool`, `can_next: bool` — можно ли листать (обновляются в `load_current` из `catalog.current_index()` и длины). Если `false` — стрелка не рисуется и клик no-op.

`scene::build` (оконная ветка): для каждой стороны, если `can_*` и `nav_alpha[i] > 0`, эмитит:
- тонкую скруглённую плашку `overlay_bg` (альфа × `nav_alpha`),
- шеврон Tabler (`chevron-left` / `chevron-right`), центрированный по высоте полосы, цвет `text_primary` × `nav_alpha`.

Глифы: `GLYPH_CHEVRON_LEFT`/`GLYPH_CHEVRON_RIGHT` (кодпоинты Tabler сверяются при реализации по `tabler-icons.css`).

### Интеграция (`app.rs`)
- `CursorMoved`: hover уже считается через `hit::hit`; `NavPrev`/`NavNext` обновляют `hovered` → запускают анимацию альфы.
- `MouseInput::Pressed`: ветки `Region::NavPrev => navigate(-1)`, `Region::NavNext => navigate(+1)` (до фото-логики), только если `can_prev`/`can_next`.

### Тесты
- `layout`: `nav_prev`/`nav_next` геометрия (x, ширина, высота = viewer.h, y = viewer.y); `ZERO` в fullscreen.
- `hit`: точка в левой/правой полосе → `NavPrev`/`NavNext`; крайние 6px → `Resize` (приоритет ресайза сохранён).
- `scene`: при hover и `can_*` эмитится шеврон нужной стороны; при `can_*=false` — не эмитится даже на hover.

## #4 — Свайп drag-жестом (только при fit)

### Поведение
- **Активация:** в `MouseInput::Pressed` ветвимся по `can_pan = zoom > min_zoom`.
  - Зум (`can_pan=true`): drag = pan (как сейчас).
  - Fit (`can_pan=false`): режим свайпа — запоминаем `swipe_press_x`, ставим `swiping=true`.
- **Перетаскивание** (`CursorMoved` при `swiping`): `dx = cursor.x - swipe_press_x`. Транзиентное горизонтальное смещение фото на `dx`.
- **Отпускание** (`Released`): если `|dx| > порог` (15% ширины viewer) и в эту сторону есть фото — листаем (`dx < 0` ⇒ next, `dx > 0` ⇒ prev), `swipe_offset` сбрасывается (новое фото грузится в fit). Иначе — плавный откат `swipe_offset → 0`.

### Смещение фото (`src/view.rs` — единственное хранилище)
Добавляем транзиентное поле `swipe_offset: f32` с `set_swipe_offset(dx)` и геттером `swipe_offset()`. В `model_matrix` прибавляется к `center.x` (`center = win*0.5 + pan + (swipe_offset, 0)`). Не клампится `clamp_pan` и переживает ежекадровый сброс `pan=ZERO` в fit-режиме (живёт отдельным полем). View остаётся чистой математикой; юнит-тест: матрица сдвигает quad на `swipe_offset`. Это **единственное** место хранения смещения — `app.rs` лишь управляет им через сеттер/геттер.

### Откат-анимация
`app.rs` держит только состояние взаимодействия: `swiping: bool`, `swipe_press_x: f32`, `swipe_anim: bool`.
- Во время drag: `view.set_swipe_offset(cursor.x - swipe_press_x)`.
- На откате: ставим `swipe_anim=true`; в `RedrawRequested` (рядом с `bottom_factor`) лерпим `view.swipe_offset()` → 0 (ease, ~0.2 с) и пишем обратно через `set_swipe_offset`; при достижении ~0 — `swipe_anim=false`.
- На листании: `set_swipe_offset(0.0)` сразу (новое фото грузится в fit).

### Чистая логика
`src/input/`:
```rust
pub enum NavDir { Prev, Next }
pub fn on_swipe_release(dx: f32, viewer_w: f32) -> Option<NavDir>;
```
Порог = `0.15 * viewer_w`. `dx < -порог` ⇒ `Next`, `dx > порог` ⇒ `Prev`, иначе `None`. Юнит-тесты (знаки, граница порога, нулевая ширина).

### Граничные случаи
- Край каталога: `on_swipe_release` вернул `Some(dir)`, но `navigate` не сдвинулся (нет фото) → откат `swipe_offset → 0`.
- Двойной клик: малое смещение → `dx < порог` → `None` → откат, навигации нет (двойной клик ловится прежней логикой).
- Конфликт с pan: исключён — pan только при зуме, свайп только при fit.

### Тесты
- `input::on_swipe_release`: `dx` за порогом влево→Next, вправо→Prev, в пределах→None, `viewer_w=0`→None.
- `view`: `set_swipe_offset(d)` сдвигает `screen_quad`/центр на `d` по X; `swipe_offset=0` эквивалентно прежней матрице.

## Тестовая стратегия

Юнит-тесты чистого ядра (input/view/layout/hit/scene) — основная гарантия. GPU/визуальная приёмка вручную: проявление стрелок на hover, инертность на краях, свайп со следованием за курсором и откатом, шорткаты на кириллической раскладке.
