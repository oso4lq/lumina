# Lumina — v0.5a: дисковый кэш миниатюр + префетч ±2 — дизайн

**Дата:** 2026-06-17
**Статус:** дизайн утверждён, готов к плану

## Проблема

Два независимых, но связанных по теме «производительность навигации» узких места:

1. **Миниатюры декодируются заново при каждом открытии папки.** `ThumbnailStore` держит
   состояния в памяти (LRU 256), текстуры — в `ThumbnailLayer`. Дисковой персистентности нет:
   при переоткрытии той же папки (или перезапуске) все миниатюры декодируются с нуля (для RAW —
   встроенное превью или полный develop, десятки МБ на кадр). На больших папках карусель «оживает»
   медленно каждый раз.

2. **Соседние полноразмерные кадры не упреждаются.** `navigate()` → `load_current()` всегда
   декодирует целевой файл с нуля (preview→full на rayon). Перелистывание вперёд/назад упирается
   в полный декод, хотя сосед почти наверняка будет показан следующим.

## Решение

Два отдельных кэш-слоя, оба — «чистое ядро + тонкая обёртка». Декод остаётся на rayon.

### Часть 1 — Дисковый кэш миниатюр (файлы на диске)

Кэш-папка `%LOCALAPPDATA%\Lumina\thumbs\` (фолбэк: `std::env::temp_dir()/lumina/thumbs`).
Одна закодированная миниатюра на файл; имя файла = стабильный хеш ключа.

- **Ключ:** `FNV-1a 64-bit` по байтам `(абсолютный путь, mtime, размер файла, целевая высота th)`
  → 16-символьный hex + `.png`. FNV-1a реализуется инлайн (тривиально, **стабилен между сборками** —
  в отличие от `std::DefaultHasher`, который для персистентности использовать нельзя). mtime+size
  в ключе дают автоматическую инвалидацию: правка файла → новый ключ → старая запись осиротеет.
- **Значение:** миниатюра в **PNG** (lossless, есть в крейте `image`, хранит размеры в заголовке,
  поддерживает альфу). Размер миниатюры мал (высота ~`THUMB_H·scale`), PNG приемлем; JPEG-вариант
  для экономии диска отмечен как возможная последующая оптимизация (YAGNI сейчас).
- **Чтение (в rayon-воркере `request_thumbnails`, ДО тяжёлого декода):** есть валидный файл кэша →
  прочитать+декодировать PNG → отдать `UserEvent::Thumbnail`. Иначе — обычный путь
  (`decode_preview`/`decode_full` → `make_thumb`) + **запись** результата в кэш (атомарно:
  во временный файл + `rename`).
- **Эвикция:** бюджет суммарного размера папки (по умолчанию **256 МБ**). Разовый фоновый
  `prune` при старте приложения (на rayon): если папка превышает бюджет — удалять файлы по
  возрастанию времени модификации, пока не уложимся. Простая и устойчивая стратегия; БД не нужна.

**Модуль `src/thumbcache.rs` (чистое ядро + тонкие IO-обёртки):**
- `cache_key(path, mtime, size, th) -> String` — чистая, юнит-тестируемая (детерминизм, разные
  входы → разные ключи, одинаковые → одинаковый).
- `cache_path(dir, key) -> PathBuf` — чистая.
- `load(dir, key) -> Option<DecodedImage>` / `store(dir, key, &DecodedImage)` — тонкие IO,
  тестируются на временной папке.
- `prune(dir, budget_bytes)` — эвикция по mtime; тестируется на временной папке.

`app.rs::request_thumbnails`/rayon-воркер используют эти функции; `make_thumb` без изменений.

### Часть 2 — Префетч ±2 полноразмерных кадров

In-memory LRU-кэш декодированных кадров, **раздельный** от дискового кэша миниатюр.

- **`PrefetchCache` (`src/prefetch.rs`, чистое ядро):** `path -> Arc<DecodedImage>`, эвикция по
  **байтовому бюджету** (сумма `rgba.len()`), не по числу. Бюджет по умолчанию **512 МБ** — так
  кэш сам ограничивается на гигантских RAW (один 40MP-кадр ≈ 160 МБ; ±2 могло бы быть >800 МБ при
  ограничении «по числу», поэтому именно байтовый бюджет). API: `get(&path) -> Option<Arc<…>>`,
  `insert(path, Arc<…>) -> Vec<PathBuf>` (вытесненные), `contains(&path)`. Юнит-тестируется на
  маленьких `DecodedImage` с подставными размерами.
- **Заполнение:** когда для текущего кадра приходит стадия `Full`, кадр кладётся в кэш, и
  запускается фоновый декод соседей (индексы ±1, ±2, которых ещё нет в кэше) на rayon. Результаты
  приходят новым `UserEvent::Prefetched { path, image }` — он **только** кладёт кадр в кэш, не
  трогает вид/генерацию. Ключ — путь (не generation), поэтому результат валиден после любой навигации.
- **Использование:** `load_current()` сперва проверяет `PrefetchCache.get(path)`. Хит → кадр
  устанавливается в рендерер немедленно (тот же путь, что обработка `Decoded{Full}`: инициализация
  вида/`inited_generation`), без спавна декода → перелистывание мгновенно. Затем — запуск префетча
  новых соседей. Промах → текущий путь (спавн preview→full), как раньше.
- **Сброс:** при смене папки (`open_file`) кэш очищается (пути другие — можно и не чистить, но
  чистим для предсказуемого расхода памяти).

## Взаимодействие и границы

- Два кэша независимы: диск — миниатюры карусели; память — полноразмерные кадры для навигации.
- Generation-guard основного декода не меняется; префетч-события не несут generation и не могут
  «показать» устаревший кадр (они лишь наполняют кэш; показ инициирует только `load_current`).
- Дисковый кэш переживает перезапуск; префетч-кэш — нет (это горячий буфер навигации).
- HEIC/RAW: миниатюра в кэше — это уже ресайзнутый RGBA из `make_thumb`, формат источника не важен.

## Тестирование

Юнит (без GPU/ФС-побочек, либо на временной папке):
- `thumbcache::cache_key` — детерминизм; чувствительность к каждому компоненту ключа; стабильный hex.
- `thumbcache::store`+`load` round-trip на временной папке (RGBA сохранён, размеры совпали).
- `thumbcache::prune` — превышение бюджета удаляет старейшие, новые остаются.
- `prefetch::PrefetchCache` — вставка/получение; эвикция по байтовому бюджету (вытесняется LRU,
  возвращается список вытесненных); `contains`.

Интеграция/ручная приёмка: открыть большую папку → миниатюры строятся; переоткрыть → мгновенно
(из кэша). Перелистывание вперёд/назад на тяжёлых RAW — соседи появляются без задержки после
первого прохода. Расход памяти не растёт неограниченно (байтовый бюджет).
