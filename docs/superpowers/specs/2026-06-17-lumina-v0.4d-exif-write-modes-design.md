# Lumina v0.4d — починка записи EXIF + режимы бэкапа/перезаписи — дизайн

**Дата:** 2026-06-17
**Статус:** утверждён (брейншторм)
**Контекст:** продолжение v0.4b (часть 2 — редактирование/запись EXIF). При ручной проверке на
реальных Fujifilm RAF обнаружен баг: путь записи для RAW (XMP sidecar) молча теряет правки.
Этот дизайн чинит запись и расширяет её двумя режимами сохранения.

---

## Проблема

Текущий `exif::write::write_edits` для RAW-форматов запускает
`exiftool <edits> -o %d%f.xmp -- file`. Эмпирически (exiftool 13.59, реальный X-T5 RAF):

1. Команда создаёт sidecar, **копируя в него исходные метаданные**, а сами правки игнорирует
   (exit 0, «1 image files created» — выглядит успехом). Правки молча теряются.
2. Даже корректная форма `-o file.xmp` пишет в sidecar **только теги группы XMP**;
   `EXIF:`/`GPS:`/`IPTC:` отбрасываются (sidecar хранит лишь XMP-namespace).
3. После сохранения popup перечитывает теги по **самому RAW**, а не по sidecar — правки не отобразятся.

Фикстурный тест части 2 (`write_real_jpg_set_and_backup`) был на JPEG (in-place, без `-o`) и эту
ветку не покрывал.

Параллельно выявлены **три разных намерения** пользователя при записи, требующие разных команд:

| Намерение | Команда exiftool | Оригинал |
|---|---|---|
| (b) Правка в файле, обратимо | in-place, бэкап `_original` | меняется, есть бэкап |
| (c) Правка/стирание необратимо (анонимизация под передачу) | `-overwrite_original` / `-all=` | затёрт, бэкапа нет |

(Sidecar-вариант (a) сознательно отвергнут: несовместим с режимом перезаписи/стирания — sidecar
ничего не удаляет из RAW, плюс требует мерджа при чтении. Решение брейншторма: **RAW пишется in-place**,
как все форматы.)

## Цель

1. **Починить запись:** все форматы (вкл. RAW) пишутся in-place; правки реально применяются и
   перечитываются.
2. **Режим бэкапа** (по умолчанию): exiftool оставляет `_original` — правка обратима (текущее поведение).
3. **Режим необратимой перезаписи** (тоггл в popup): `-overwrite_original`, без бэкапа.
4. **Действие «Стереть всё»** (доступно только в необратимом режиме): удаляет все метаданные
   (`-all=`), сохраняя Orientation + ICC, чтобы фото отображалось верно и с правильным цветом.

Бандл exiftool остаётся в v0.5. Sidecar как отдельная фича — не входит в scope (при желании — будущая версия).

## Не-цели

- Sidecar для RAW (отвергнут).
- Затирание свободного места диска / форензическая необратимость на стороне автора (вне модели угроз:
  цель — чтобы **получатель файла** не восстановил исходные метаданные; `-overwrite_original` это обеспечивает,
  так как получателю достаются только затёртые байты).
- Дедуплицированный бэкап / история версий (exiftool создаёт `_original` один раз — пристинная версия).

## Архитектура

Сохраняется разделение «чистое ядро ↔ тонкие обёртки» из части 2. Построение аргументов exiftool —
**чистые функции** (юнит-тестируемы); запуск subprocess — обёртка без логики. UI (`scene`/`hit`/`layout`)
тестируется без GPU. Новых GPU-пайплайнов нет — всё на существующих `DrawCmd`.

### 1. `exif::write` — запись с режимом + стирание

```rust
/// Режим записи: оставлять ли восстановимый бэкап.
#[derive(Clone, Copy, Debug, PartialEq, Eq)]
pub enum WriteMode {
    /// Обычный: exiftool делает `_original` (обратимо). По умолчанию.
    Backup,
    /// Необратимый: `-overwrite_original`, без бэкапа.
    Overwrite,
}

/// Аргументы exiftool для набора правок (без пути): edits_to_args(edits) + флаг режима.
/// Чистая функция — юнит-тест.
pub fn edit_args(edits: &[TagEdit], mode: WriteMode) -> Vec<String>;

/// Аргументы стирания всех метаданных с сохранением Orientation + ICC (всегда необратимо).
/// Чистая функция — юнит-тест. Возвращает:
/// ["-all=", "-tagsfromfile", "@", "-orientation", "-icc_profile", "-overwrite_original"]
pub fn strip_args() -> Vec<String>;

/// Применить правки in-place (все форматы). Пустой список — no-op.
pub fn write_edits(path: &Path, edits: &[TagEdit], mode: WriteMode) -> Result<()>;

/// Стереть все метаданные (Orientation + ICC сохраняются), необратимо.
pub fn strip_all(path: &Path) -> Result<()>;
```

- Ветка `RawDecoder::supports → -o %d%f.xmp` **удаляется**. `write_edits` строит
  `exiftool <edit_args(edits, mode)> -- path`.
- `WriteMode::Backup` → без доп.флага (exiftool оставляет `_original`).
- `WriteMode::Overwrite` → добавляется `-overwrite_original`.
- `strip_all` → `exiftool <strip_args()> -- path` (всегда `-overwrite_original`).
- Разделитель `--` сохраняется (защита от подмены флагов именем файла).
- Обработка ошибок — как сейчас (`LuminaError::Exif(path, stderr)`).

**Проверено эмпирически (exiftool 13.59, реальный RAF):**
- in-place EXIF/XMP/IPTC/GPS + `-gps:all=` — пишутся и перечитываются;
- `-all= -overwrite_original` — Make/Model/Artist/SerialNumber/DateTimeOriginal/Software снесены,
  `_original` не создан, встроенное превью тоже чисто; остаётся лишь структурно-сенсорная инфа RAF.

### 2. `exif::tags`
Без изменений API. `edits_to_args` (часть 2) переиспользуется внутри `edit_args` (она же добавляет
флаг режима).

### 3. UI — `theme` / `layout` / `scene` / `hit`

**theme:** цвет `danger` уже есть (часть 2). При необходимости — `danger_bg` (фон активного
необратимого тоггла/кнопки «Стереть всё»). Глиф чекбокса — из существующих шрифтов (Segoe MDL2:
`\u{E73A}` checkbox-off / `\u{E73D}` checkbox-on — уточнить кодпоинты в реализации) либо рисуем
рамку+галочку примитивами.

**layout:** футер уже есть (`POPUP_FOOTER_H`, `popup_footer_buttons`). Добавить геометрию:
- зона тоггла слева (`popup_footer_toggle(p, scale) -> Rect`);
- кнопка «Стереть всё» (`popup_footer_strip(p, scale) -> Rect`) — между тогглом и `[Отмена]`,
  ненулевая только когда показывается.
Раскладка футера (необратимый режим вкл): `[☑ Необратимо]  …  [Стереть всё] [Отмена] [Сохранить]`.
Подписи компактные. Точные ширины/перенос — добиваются в реализации (ширины карточки 560 хватает).

**scene:**
- `PopupEditState` дополняется: `overwrite_mode: bool`, `confirm: ConfirmKind`.
- `build_popup` рисует тоггл (вкл — danger-подсветка + галочка) и, при `overwrite_mode`, кнопку
  «Стереть всё» (danger).
- Confirm-бар обобщается: вместо булева `confirm_close` — `ConfirmKind`:
  ```rust
  pub enum ConfirmKind { None, CloseWithPending, OverwriteSave, StripAll }
  ```
  Тексты бара:
  - `CloseWithPending` — «Несохранённые изменения — Сохранить / Отменить / Продолжить» (как сейчас);
  - `OverwriteSave` — «Необратимо, бэкапа не будет — Перезаписать / Отмена»;
  - `StripAll` — «Удалить ВСЕ метаданные, необратимо — Стереть / Отмена».

**hit:** `hit_popup_edit` получает регионы:
- `FooterToggle` (клик по тогглу необратимости);
- `FooterStrip` (кнопка «Стереть всё», только в необратимом режиме);
- обобщённые confirm-регионы: `ConfirmPrimary` / `ConfirmSecondary` / `ConfirmTertiary`
  (вместо нынешних `ConfirmSave`/`ConfirmDiscard`/`ConfirmKeep`) — app трактует их по текущему `ConfirmKind`.
  Существующие `FooterSave`/`FooterCancel`/`RowEdit`/`RowDelete`/`GpsDeleteAll` сохраняются.

### 4. `app.rs` — состояние и поток

**Состояние** (замена/добавление к части 2):
- `exif_overwrite_mode: bool` — состояние тоггла;
- `exif_confirm: ConfirmKind` — вместо `exif_confirm_close: bool`;
- `exif_pending_strip_all: bool` — staged-флаг полного стирания (эксклюзивен: при нём построчные
  правки игнорируются).
Все сбрасываются при открытии popup (тоггл — всегда выкл, безопасность).

**Поток кликов:**
- Тоггл → `exif_overwrite_mode = !exif_overwrite_mode` (выключение сбрасывает `exif_pending_strip_all`).
- «Стереть всё» (только в необратимом) → `exif_confirm = StripAll`.
- Save:
  - необратимый режим **и** есть pending-правки → `exif_confirm = OverwriteSave`;
  - обычный режим → `write_edits(Backup)` сразу (как сейчас).
- Confirm-бар (по текущему `ConfirmKind`):
  - `OverwriteSave` → primary: `write_edits(Overwrite)`; secondary: отмена бара;
  - `StripAll` → primary: `strip_all`; secondary: отмена;
  - `CloseWithPending` → как в части 2 (Сохранить→save в текущем режиме / Отменить→discard+close / Продолжить).
- Закрытие с pending-правками — как в части 2 (бар `CloseWithPending`). Если включён необратимый режим,
  «Сохранить» из этого бара тоже идёт через `OverwriteSave`-подтверждение (двойное подтверждение
  допустимо и безопасно).

**Запись** — на rayon, через `UserEvent::ExifSaved` (как часть 2). `exif_save` принимает текущий
`WriteMode`; добавляется `exif_strip_all` (зовёт `strip_all`). После успеха — перечитывание тегов,
очистка pending/strip/confirm, сброс `exif_overwrite_mode` в выкл (безопасность: следующее сохранение
снова осознанное). Заголовок обновляется при смене Model (как сейчас).

### 5. Поток данных

```
клик тоггла → overwrite_mode
клик «Стереть всё» → confirm=StripAll → (бар) primary → rayon: strip_all → ExifSaved → перечитать
Save (overwrite) → confirm=OverwriteSave → (бар) primary → rayon: write_edits(Overwrite) → ExifSaved → перечитать
Save (backup) → rayon: write_edits(Backup) → ExifSaved → перечитать
```

### 6. Обработка ошибок
Без изменений: stderr exiftool → `LuminaError::Exif` → баннер `exif_error`, правки/состояние не теряются.

## Тестирование

**Юнит (чистые функции, без GPU/subprocess):**
- `exif::write`: `edit_args(edits, Backup)` == `edits_to_args(edits)`; `edit_args(edits, Overwrite)`
  добавляет `-overwrite_original`; `strip_args()` == ожидаемый вектор; пустые правки.
- `ui::layout`: геометрия тоггла и кнопки «Стереть всё» (внутри футера; strip правее тоггла, левее Отмены).
- `ui::scene`: тоггл рисуется; в `overwrite_mode` есть кнопка «Стереть всё»; три варианта confirm-бара
  дают ожидаемые тексты; pending-маркеры не ломаются.
- `ui::hit`: `FooterToggle`/`FooterStrip` по координатам; confirm-регионы; strip-кнопка не кликается
  вне необратимого режима.

**Интеграционные `#[ignore]` (нужен exiftool, реальный файл):**
- `write_overwrite_no_backup`: правка в Overwrite → значение записано, `_original` **отсутствует**.
- `strip_all_removes_pii_keeps_orientation`: после `strip_all` — GPS/Artist/Serial/DateTimeOriginal
  отсутствуют, Orientation и ICC присутствуют.

Существующий `write_real_jpg_set_and_backup` адаптируется под новую сигнатуру (`WriteMode::Backup`).

## Совместимость и риски
- Сигнатура `write_edits` меняется (+`WriteMode`) — единственный вызов в `app.rs`; правится там же.
- Необратимая запись/стирание — деструктивны: защита через явный confirm-бар (Q3) + danger-стиль +
  сброс тоггла после каждого сохранения.
- in-place меняет байты RAW (хэш/дата) даже в режиме Backup — это осознанное следствие отказа от sidecar
  (зафиксировано в брейншторме).

## Порядок реализации
- **Фаза 1 — починка записи:** RAW → in-place (`write_edits` без `-o`), `WriteMode` (Backup по умолчанию),
  адаптация вызова в `app.rs` и теста. Отдельный коммит — закрывает баг v0.4b.
- **Фаза 2 — режимы и стирание:** тоггл необратимости, `strip_all`/«Стереть всё», обобщённый
  `ConfirmKind`, UI/hit/layout, подтверждения, интеграционные тесты.

Один спек, один план; задачи последовательны (фаза 1 перед фазой 2).
