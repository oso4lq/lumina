# Lumina — v0.5c: свайп трекпадом (горизонтальная навигация) — дизайн

**Дата:** 2026-06-17
**Статус:** дизайн утверждён, готов к плану

## Проблема

В v0.4c добавлен свайп **drag-жестом** (зажать ЛКМ и тянуть при fit) — `input::on_swipe_release`.
Но навигация **двухпальцевым горизонтальным жестом тачпада** не работает: обработчик
`WindowEvent::MouseWheel` использует только вертикальную компоненту (`PixelDelta.y` → зум,
`LineDelta.y` → зум; над каруселью — горизонтальный скролл ленты). Горизонтальная компонента
`PixelDelta.x` (которую тачпад шлёт при двухпальцевом смахивании влево/вправо) игнорируется в зоне
viewer. Пользователи тачпада ожидают листать фото смахиванием, как в нативных просмотрщиках.

## Решение

В зоне viewer (не над каруселью, не в EXIF popup) обрабатывать **горизонтально-доминирующий**
`PixelDelta` как пошаговую навигацию по каталогу: накапливать `dx`, при превышении порога —
шаг prev/next, порог вычитается из аккумулятора (остаток переносится). Тип события различается:
`PixelDelta` с `|dx| > |dy|` → навигация; всё прочее (вертикаль, `LineDelta` мышиного колеса) —
прежнее поведение (зум/скролл карусели).

### Чистое ядро (`src/input/mod.rs`)

Аккумулятор-логика — чистая функция, юнит-тестируемая:
```rust
/// Порог накопленного горизонтального смещения тачпада (в логических px) на один шаг навигации.
pub const TRACKPAD_NAV_STEP_PX: f32 = 80.0;

/// Накопить горизонтальное смещение тачпада и решить, листать ли.
/// `accum` — мутируемый аккумулятор (px). Возвращает число шагов со знаком:
/// >0 — вперёд (next) N раз, <0 — назад (prev) |N| раз, 0 — пока недостаточно.
/// Жест считается горизонтальным только если |dx| >= |dy| (иначе это вертикальная прокрутка).
pub fn on_trackpad_pan(accum: &mut f32, dx: f32, dy: f32, step_px: f32) -> i32 { ... }
```
Поведение:
- Если `dx.abs() < dy.abs()` → вернуть 0, аккумулятор **сбросить в 0** (переключились на вертикаль).
- Иначе `*accum += dx`; вычислить `steps = (*accum / step_px).trunc()`; уменьшить `*accum` на
  `steps * step_px`; вернуть знак-инвертированный шаг навигации.
- **Знак:** смахивание влево (контент уходит влево, `dx < 0`) на тачпаде = «следующий» — как лента
  фото. То есть `dx < 0 → next (+1)`, `dx > 0 → prev (−1)`. Это согласуется с `on_swipe_release`
  (тянули влево → Next). Возвращаем `-steps` (т.к. `dx<0` даёт отрицательные `steps`, инвертируем в +1=next).

> Уточнение знака реализуется так, чтобы один «полный» свайп влево = один `next`. Зум-направление
> (вертикаль) не затрагивается.

### Обёртка (`src/app.rs`)

- Поле состояния `trackpad_accum: f32` в `AppState` (сброс при навигации/смене папки не обязателен —
  логика сама вычитает порог).
- В `WindowEvent::MouseWheel`:
  - Сохранить «сырой» `delta` для определения типа.
  - В EXIF popup и над каруселью — поведение без изменений (ранние ветки уже есть).
  - В зоне viewer: если `delta` — `PixelDelta` → вызвать `on_trackpad_pan(&mut accum, dx, dy, STEP)`;
    при ненулевом результате — выполнить `navigate(±1)` нужное число раз (или один раз, ограничив
    шаг ±1 за событие для предсказуемости — см. границы). Если шагов нет и жест вертикальный —
    обычный зум по `y` (как сейчас).
  - `LineDelta` (мышиное колесо, дискретное, без `x`) → зум по `y`, как сейчас (тачпад-навигацию не
    триггерит).

### Границы (осознанно)

- **Только `PixelDelta`** (тачпад/высокоточные устройства) триггерит навигацию. Мышиное колесо
  (`LineDelta`) — всегда зум: у него нет горизонтальной оси, и пользователи мыши не ждут листания колесом.
- **Один шаг за событие максимум** (клампим результат до ±1), чтобы инерционный «бросок» тачпада не
  пролистывал десяток кадров разом. Остаток аккумулятора переносится — быстрый повтор жеста листает
  дальше естественно. (Накопитель и порог делают это плавным.)
- **Не конфликтует с зумом:** разделение по доминантной оси (`|dx|` vs `|dy|`). Диагональные шумы
  гасятся сбросом аккумулятора при вертикальном доминировании.
- **Карусель/попап** перехватывают колесо раньше (существующие ветки) — там тачпад-навигация не работает.

## Тестирование

Юнит (`src/input/mod.rs`, чистые):
- Накопление ниже порога → 0 шагов, аккумулятор растёт.
- Преодоление порога влево → +1 (next); вправо → −1 (prev); остаток переносится.
- Вертикальное доминирование (`|dy|>|dx|`) → 0 и сброс аккумулятора.
- Большой бросок (например 3×порог за раз) → функция возвращает 3, но обёртка клампит до 1
  (тест клампа — на стороне ядра как отдельная проверка значения, кламп в app.rs покрывается ручной приёмкой).

Ручная приёмка (на ноутбуке с тачпадом): двухпальцевое смахивание влево/вправо в зоне фото листает
вперёд/назад по одному кадру; вертикальный двухпальцевый жест — зум; над каруселью — скролл ленты;
мышиное колесо — зум.
