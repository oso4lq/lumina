# Lumina v0.1 — дизайн (Viewer core)

Дата: 2026-06-15
Статус: одобрен к реализации
Базовая спецификация: `SPEC_lumina.md` (v0.2)

## Цель v0.1

Минимальное работающее ядро просмотрщика (§12 спеки):

- [ ] winit-окно + wgpu-контекст
- [ ] Загрузка и показ JPEG/PNG (и прочих форматов, что `image-rs` тянет даром)
- [ ] Zoom (колёсико) и pan (drag)
- [ ] Double-click fit ↔ 100%
- [ ] Навигация стрелками по файлам папки

Всё остальное из спеки (RAW, HEIC, EXIF, карусель, titlebar, slideshow, темизация, installer)
осознанно отложено в фазы v0.2–v0.6.

## Архитектурный подход

**Монолитный `App` + immediate-mode рендер**, с trait-границами под будущие форматы.

- Одна структура `App`, реализующая `winit::ApplicationHandler` (модель winit 0.30).
- Всё состояние (`AppState`) живёт только в главном потоке; межпоточно ходят лишь готовые
  результаты декодирования через `EventLoopProxy`.
- Каждый кадр заново вычисляем матрицу трансформации из `ViewTransform` и рисуем textured quad.
- Декодирование уходит в rayon (CPU-bound), без tokio.

Отвергнуто: шина сообщений между подсистемами (преждевременная сложность для 5 фич),
scene-graph/ECS (overkill для одного quad), UI-фреймворки egui/iced (противоречат концепции
«всё рисуем сами через wgpu, frameless-окно»).

### Trait-границы (задел без лишнего кода в v0.1)

```rust
pub trait Decoder {
    fn supports(ext: &str) -> bool where Self: Sized;
    fn decode(&self, path: &Path) -> Result<DecodedImage>;
}
```

- v0.1: одна реализация `StandardDecoder`.
- Позже `RawDecoder` / `HeicDecoder` встают без изменения вызывающего кода.
- `Renderer` имеет узкий API (`upload_texture`, `render`, `resize`) — внутренности wgpu
  можно менять, не трогая `app.rs`.

## Поток данных и декодирование

Жизненный цикл открытия фото:

```
1. App получает индекс файла (старт / стрелка / drag&drop / диалог)
2. AppState.pending_load = Some(generation_id)        // монотонный u64-счётчик
3. rayon::spawn: StandardDecoder.decode(path) на фоновом потоке
       └─ Result<DecodedImage { rgba, width, height }>
4. По готовности фон шлёт через EventLoopProxy:
       UserEvent::Decoded { generation_id, result }
5. Event loop ловит событие:
       - если generation_id != текущего → отбрасываем (юзер уже листнул дальше)
       - иначе: Renderer.upload_texture(rgba) → AppState.texture, request_redraw
```

**`generation_id`:** при быстром листании rayon-задачу не отменяем (она уже бежит), но
игнорируем устаревший результат по монотонному счётчику. Без отмены, без гонок, без мьютексов
на состоянии.

**Связь фон → UI:** `EventLoop::create_proxy()` + `EventLoopProxy::send_event(UserEvent)` —
штатный способ winit будить event loop из чужого потока.

```rust
pub struct DecodedImage {
    pub rgba: Vec<u8>,     // RGBA8
    pub width: u32,
    pub height: u32,
}
```

Не в v0.1: префетч соседних ±2 (v0.5), LRU-кэш текстур, embedded preview. Один файл за раз.
Между запуском декода и готовностью на экране остаётся прошлая текстура (или фон `#111113` на
первом запуске). Спиннеров нет — декод JPEG/PNG быстрый.

## ViewTransform — zoom/pan и анимация

```rust
pub struct ViewTransform {
    zoom: f32,            // 0.05 .. 32.0 (5%..3200%)
    pan: Vec2,            // смещение в пикселях экрана
    is_fit: bool,         // режим fit «прилипает» к ресайзу окна
    // анимация zoom
    zoom_target: f32,
    zoom_anim_start: f32,
    anim_elapsed: f32,    // секунды; >= 0.15 → анимация закончена
    // заложено под v0.4, в v0.1 не используется:
    rotation: u8,
    flip_h: bool,
    flip_v: bool,
}
```

- **Fit-to-window** — вычисляемый, не хранимый: `fit_zoom = min(win_w/img_w, win_h/img_h)`.
  При ресайзе окна пересчитывается сам, пока `is_fit == true`.
- **100% (1:1)** — `zoom = 1.0`.
- **Zoom с центром под курсором:** `pan' = cursor - (cursor - pan) * (zoom_new / zoom_old)`.
- **Double-click** переключает `is_fit`: fit ↔ 100%, с анимацией к новому zoom.
- **Анимация:** при смене целевого zoom ставим `zoom_target`, обнуляем `anim_elapsed`,
  флаг `animating`. В redraw продвигаем `anim_elapsed += dt`, считаем
  `zoom = lerp(start, target, ease_out_cubic(t))`, `t = elapsed / 0.15`. Пока `t < 1.0` —
  `request_redraw`. Pan при drag — мгновенный, без анимации.
- **Матрица:** ортопроекция «изображение в пикселях → clip space» со вшитыми zoom и pan,
  считается через `glam::Mat4` каждый кадр из `ViewTransform` + размеров окна и изображения.
  Шейдер `blit.wgsl` (§9 спеки) берёт её как uniform.
- **Pan-clamp:** при `zoom > fit` изображение не «улетает» за разумные границы; при
  `zoom <= fit` картинка центрируется, pan игнорируется.
- **dt:** время кадра считаем через `std::time::Instant` между кадрами.

## Ввод и навигация

Подмножество §5 спеки — только viewer-ядро:

| Ввод | Действие |
|---|---|
| Колёсико ↑/↓ | Zoom in/out с центром под курсором |
| Double-click | Fit ↔ 100% (с анимацией) |
| Ctrl+0 | Fit to window |
| Ctrl+1 | 100% |
| Drag (при zoom > fit) | Pan; курсор → grab/grabbing |
| `→` / `←` | Следующее / предыдущее фото |
| `Home` / `End` | Первое / последнее |
| Перетаскивание файла в окно | Открыть файл + пересканировать папку |

Отложено: F/F11, R, Shift+R, H, V, I, Delete, Ctrl+C, Ctrl+Z, Esc, Space — в свои фазы.

**Модуль `input`:** чистая функция-роутер `handle_event(&mut AppState, &WindowEvent) ->
InputOutcome`, где `InputOutcome` сообщает app, нужно ли `request_redraw` и/или запустить
загрузку нового файла. Мутации состояния в одном месте, логика тестируема без окна.

**`FolderCatalog`:**
```rust
pub struct FolderCatalog {
    files: Vec<PathBuf>,    // отсортированы natord по имени (name_asc)
    current: usize,
}
```
При открытии файла: берём папку, собираем файлы с поддерживаемыми расширениями
(`StandardDecoder::supports`), сортируем natord, находим позицию открытого файла.
`next()/prev()` — насыщающие (упираются в края, без зацикливания). Прочие режимы сортировки
(§16) — v0.6.

**Double-click vs drag:** двойной клик — по времени между нажатиями (~400 мс) и малому
смещению; зажатие+перетаскивание — pan.

## Окно

Для v0.1 — **обычное декорированное окно** (`with_decorations(true)`) с системным заголовком
(`window.set_title("имя_файла — Lumina")`). Кастомный frameless-titlebar (§4.1) с ручным
hit-testing'ом и кнопками управления — фаза v0.3, чтобы не раздувать v0.1 косметикой.

## Обработка ошибок

Тип `LuminaError` через `thiserror`. Принцип: **приложение не падает из-за одного плохого файла.**

| Ситуация | Поведение |
|---|---|
| Файл не декодируется (битый/неподдерживаемый) | Лог `warn`, на экране остаётся прошлая текстура + текст-оверлей «Не удалось открыть файл»; навигация работает |
| CLI-арг — несуществующий путь | Открываем диалог выбора файла |
| Запуск без арга | Диалог `rfd`; отмена → пустой фон `#111113`, ждём drag&drop |
| Папка пуста / один файл | Навигация упирается в края |
| Фатальные (нет wgpu-адаптера, не создаётся окно) | `Result` из инициализации → лог `error` + аккуратное завершение с сообщением |

Декод на rayon возвращает `Result<DecodedImage>`; ошибка прилетает тем же
`UserEvent::Decoded` и обрабатывается в главном потоке — паника фона не роняет приложение.

## Тестирование

GUI/wgpu-цикл юнит-тестами не покрываем. Покрываем чистую логику по TDD (тест → реализация):

- **`view.rs`** (основной кандидат на TDD): zoom-clamp в 5%–3200%; «точка под курсором не
  двигается» при zoom; fit-zoom при разных соотношениях сторон; pan-clamp; ease-функция
  (граничные t=0, t=1).
- **`catalog`**: natord-сортировка (`IMG_2 < IMG_10`); фильтрация по расширению; поиск позиции
  открытого файла; насыщение `next/prev` на краях.
- **`decoder`**: роутер по расширению (`supports`); декод тестового PNG/JPEG из
  `tests/fixtures` → проверка размеров и длины RGBA-буфера.
- **`input`**: на чистой функции-роутере — колёсико меняет zoom; стрелка просит загрузку
  соседнего индекса; double-click переключает `is_fit`.

`Renderer` и `App` — ручная проверка через `cargo run` на реальных фото.

### Чек-лист ручной приёмки v0.1

1. `cargo run -- путь\к\фото.jpg` открывает окно с фото, вписанным в окно (fit).
2. Колёсико зумит к курсору; зум плавный (~150 мс).
3. Double-click переключает fit ↔ 100% с анимацией.
4. Ctrl+0 → fit, Ctrl+1 → 100%.
5. При zoom > fit drag двигает изображение; курсор меняется на grab/grabbing.
6. `→`/`←` листают фото в папке в natord-порядке; `Home`/`End` — края.
7. Перетаскивание файла в окно открывает его и пересканирует папку.
8. Запуск без аргумента показывает системный диалог выбора файла.
9. Битый/неподдерживаемый файл не роняет приложение (лог + оверлей).
10. Ресайз окна сохраняет fit-режим (картинка остаётся вписанной).
