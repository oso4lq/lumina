# Lumina — очистка IFD1-дублей при записи EXIF — дизайн

**Дата:** 2026-06-17
**Статус:** дизайн утверждён, готов к плану

## Проблема

При редактировании EXIF-тега, который камера зеркалит в IFD1 (thumbnail-IFD), правка
Lumina записывается в IFD0 корректно, но при перечитывании **не видна в UI**.

Конкретика (воспроизведено на реальном Fuji X-Trans RAF, exiftool 13.59):
- Fuji дублирует `Artist` в IFD0 **и** IFD1.
- Запись Lumina `-EXIF:Artist=value` обновляет только IFD0 (это поведение exiftool по умолчанию — preferred-группа).
- Команда чтения Lumina `exiftool -json -G` (family 0) схлопывает `IFD0:Artist` и `IFD1:Artist`
  в один ключ `"EXIF:Artist"` и отдаёт **IFD1** (он читается последним, last-wins).
- Итог: пользователь правит Artist → сохраняет → при перечитывании видит старое (камерное)
  значение из IFD1, будто правка снова потеряна. Это тот же симптом, что баг sidecar из v0.4b,
  но по другой причине — на read-пути.

Масштаб узкий: затрагивает только EXIF-теги, которые конкретная камера дублирует в IFD1
(чаще всего `Artist`, иногда `Copyright`/`ImageDescription`/`DateTime`). Прочие EXIF-теги
(напр. `ImageDescription`, если он не в IFD1) и все XMP/IPTC/GPS-теги перечитываются верно.
Это **не регрессия v0.4d** — команда чтения не менялась с v0.4b ч.1; v0.4d лишь сделал
запись RAW рабочей, отчего проблема стала заметной именно на сценарии «правка → перечитывание».

## Решение (write-side)

При формировании аргументов exiftool для правки EXIF-группового тега дополнительно
**очищать его IFD1-дубль**: к существующему аргументу дописывать `-IFD1:{tag}=`
(удаление thumbnail-копии). После этого тег остаётся только в IFD0 — авторитетном —
и чтение `-json -G` возвращает именно отредактированное значение.

Проверено end-to-end на реальном RAF: `-EXIF:Artist=Val -IFD1:Artist=` →
`-json -G` отдаёт `Val`; встроенное превью и thumbnail целы.

### Где меняется

Единственная правка — чистая функция `exif::tags::edits_to_args` (`src/exif/tags.rs`).
Для каждой правки:
- `Set { group: "EXIF", tag, value }` → `["-EXIF:{tag}={value}", "-IFD1:{tag}="]`
- `Delete { group: "EXIF", tag }` → `["-EXIF:{tag}=", "-IFD1:{tag}="]`
- любая правка с группой **не** `EXIF` (`XMP`/`IPTC`/`GPS`) → как прежде, один аргумент
  (IFD1 — это EXIF/TIFF-концепт, у этих групп дублей в IFD1 нет)
- `DeleteAllGps` → как прежде (`-gps:all=`)

Сигнатура `edits_to_args` не меняется (`&[TagEdit] -> Vec<String>`), но теперь возвращает
переменное число аргументов на правку (был ровно один). Вызывающий код (`write::edit_args`,
добавляющий `-overwrite_original` в конец) от этого не страдает — он работает с плоским
вектором.

### Почему generic и безопасно

- `-IFD1:{tag}=` на теге, которого в IFD1 нет, — no-op-удаление (exiftool не находит и не
  ошибается; подтверждено). Поэтому правило применяется ко всем EXIF-тегам безусловно, без
  «знания» о конкретных дублях.
- На форматах без IFD1 (или без thumbnail) — тоже no-op. RAF-специфики в коде нет.
- Lumina редактирует только PII/пользовательские теги (WRITABLE_GROUPS). IFD1 легитимно
  содержит лишь структуру thumbnail (Compression/Offset/Length/Resolution), которую Lumina
  не трогает. Удаление IFD1-копии Artist/Copyright/… всегда корректно — это устаревшее зеркало.

### Граница решения (осознанно)

Фикс действует **в момент записи**. RAF, который не редактировали, при просмотре по-прежнему
покажет IFD1-значение неотредактированного дублированного тега — но это та же камерная
величина, что и в IFD0, поэтому визуального расхождения нет. Полная нормализация чтения
(read-side приоритет IFD0) намеренно отвергнута как инвазивная (ломает family-0 модель групп
и read-путь v0.4b) ради узкой проблемы.

## Тестирование

Юнит-тесты (`src/exif/tags.rs`, без GPU/exiftool):
- `Set`/`Delete` группы `EXIF` дают два аргумента, второй — `-IFD1:{tag}=`.
- `Set`/`Delete` групп `XMP`/`IPTC`/`GPS` дают один аргумент (без IFD1-очистки).
- `DeleteAllGps` — без изменений.
- Существующий `edits_to_args_set_delete_gps` обновляется под новый вывод для EXIF-правки.

Интеграционный (`#[ignore]`, реальный exiftool, на копии фикстуры): подготовить тег в IFD0+IFD1,
записать через `write_edits`, прочитать `-json -G` — значение совпадает с записанным
(дубль IFD1 не «перебивает»). Опционально: проверить на dev-shim.

Ручная приёмка: на копии Fuji RAF — правка Artist → Save → значение осталось при перечитывании.
